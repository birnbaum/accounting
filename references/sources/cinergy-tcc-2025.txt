A. Worst Case Power Consumption
   In this subsection, we describe how to model the base power
consumption used in the C INERGY framework. Specifically,
we aim to estimate the power consumption of a VM running
in isolation—that is, on a dedicated host—where all system
power can be attributed to the single workload. This situation
is referred to as the worst case, as servers are typically more
energy efficient at higher usage [34].
   To address this, we identify three key research questions:
  1) How can WCPC be estimated?
  2) How should static power consumption be distributed?
  3) How can WCPC be applied in real-world scenarios?
   We answer these questions in the following subsections and
discuss the design choices made in the framework.

D. Discussion & Limitations
   We evaluated WCPC accuracy using a model based solely
on CPU time and RAPL measurements, as these are commonly
used by existing power profiling tools. While this approach
achieves good accuracy on the evaluated generic workloads,
it has important limitations. In particular, identical CPU time
values can correspond to different power consumptions de-
pending on the instruction mix and memory behavior (e.g.,
scalar versus vector units, cache intensity), which can bias
estimates for specific workloads. In addition, RAPL cap-
tures only socket-level components (cores, uncore, cache)
and excludes other resources such as network and storage,
leading to an underestimation of total consumption. More fine-
grained models incorporating additional performance counters
or resource-specific measurements could therefore improve
WCPC accuracy for heterogeneous workloads.
   Our evaluation used production-class servers to ensure a
measurable baseline for model accuracy, as isolated software
power consumption can be directly measured—unlike bottom-
up methods that infer it indirectly. This setup emphasizes
higher power values due to the servers’ static consumption.
